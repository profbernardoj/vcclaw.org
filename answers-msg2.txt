## Q5: Context being lost between sessions that needs fixes

**1. Cross-session project status isn't captured systematically.** Each session picks up from MEMORY.md + daily files, but there's no "current project status" snapshot. Example: the 28 flavor websites are all live, but 4 CTA integration tracks are blocked — this was buried in a daily file rather than tracked prominently. Result: have to rediscover blockers each session.

**Fix:** Add "Active Blocked Items" section to MEMORY.md with explicit owner/timeline for each blocker. Or create `memory/projects/active-blockers.md` that the session reads automatically.

**2. Cron job outputs don't surface to main session.** The venice-key-monitor, venice-402-watchdog, and finance-tracker all run on schedules. Their outputs go to log files or memory/Finance.md. But when a key drops or a payment fails, there's no proactive signal to the main session. Result: key3 ran dry and wasn't noticed until the 402 watchdog caught it.

**Fix:** Cron jobs should append to a `memory/cron-alerts.md` that HEARTBEAT.md checks. Or use session_send to the main session when critical thresholds are crossed.

**3. Meeting notes are scattered.** David shared context from Morpheus maintainers meetings, BasedAI discussions, IronHill negotiations — but these live in daily files, not in structured project memory. When David asks "what did Thomas say about X", finding it requires searching through unstructured daily logs.

**Fix:** After each meeting mentioned, create a structured `memory/meetings/YYYY-MM-DD-meeting-topic.md` with attendees, decisions, action items. Reference from project files.

**4. Codebase changes between sessions lose context.** Session A modifies openclaw.json or a script. Session B doesn't know about it. The git diff shows the change, but there's no "why" attached. Result: can't tell if a setting is intentional or abandoned experiment.

**Fix:** Any non-trivial config change should be logged in `memory/config-changelog.md` with date, reason, and expected impact. This already exists for some items but isn't systematic.

**5. Pain points are forgotten after they're "solved".** Lessons-learned.md exists, but it only captures technical gotchas. The pattern of "David had to remind me X three times" doesn't get recorded. Result: same friction repeats.

**Fix:** After each weekly review, capture 1-2 process friction points in `memory/reference/process-lessons.md` with the behavioral change needed.
