## Q12: Repeated errors and guardrails to prevent them

**1. Git working directory drift — leaving uncommitted changes.**

Pattern: I modify files (SOUL.md, scripts, memory/) during a session but don't commit them. When the next session starts, `git status` shows a mess. Unclear what's intentional vs experimental. Causes confusion about "what changed" and makes rollbacks harder.

Guardrail: Night shift should commit "WIP: session context updates" at the end of every cycle. Day shift should commit before each HEARTBEAT_OK. Never leave the workspace with uncommitted changes across shift boundaries.

Enforcement: Add a check to shifts/context.md or a pre-cycle hook: `git diff-index --quiet HEAD -- || echo "ERROR: uncommitted changes, must commit before proceeding"`

**2. Cached numbers going stale — citing outdated values without refreshing.**

Pattern: I cite "$31.1M" from Feb 22 without checking if the market moved. I cite "key3 has 82 DIEM" when it's now at 0.91 USD. The numbers in memory files drift from reality.

Guardrail: Any financial number must be refreshed before citing. If a cron fetches it, check the cron's last run time. If last run was >24h ago, trigger a refresh. For market-dependent values (token prices, USD balance), always include a timestamp: "$31.1M as of 2026-02-22" or re-fetch.

Enforcement: When finance-tracker runs, it should update a `memory/finance-snapshot.md` with timestamp. Any answer citing finances must include `(as of TIMESTAMP)` or refresh first.

**3. File path assumptions — assuming a file exists without checking.**

Pattern: I edit a file like `memory/projects/everclaw/architecture.md` assuming it exists because I remember creating it. If the session context is wrong, the edit fails. I've had to recreate files that I thought existed.

Guardrail: Before editing any file not in shifts/tasks.md explicit path list, run `read` first. If it doesn't exist, error out clearly rather than silently creating a new file with different structure.

Enforcement: For memory files beyond the core (MEMORY.md, daily/, shifts/), always read-before-edit. New file creation should be explicit with clear reason.

**4. Cron job config changes without testing the cron.**

Pattern: I modify a script (venice-402-watchdog.sh) but don't actually trigger the cron to test it. The script runs hours later, fails, and I discover it after damage is done (or after multiple failures).

Guardrail: Any cron script modification must include a manual test (`./script.sh` or equivalent) before marking done. If the script requires specific conditions (e.g., 402 error), simulate the condition or wait for the cron to run before marking done.

Enforcement: In shifts/tasks.md steps that modify crons, add explicit "Test: run script manually and verify output" as part of the step.

**5. Forgetting to update shifts/state.json after completing a step.**

Pattern: I complete a step, mark it [x] in tasks.md, but forget to increment cyclesRun in state.json. The next cycle sees stale cycle count and may double-execute or misreport progress.

Guardrail: After any tasks.md edit, immediately edit state.json. Don't batch them. The state.json update is the commit of the step.

Enforcement: The cycle executor algorithm should enforce this. Add to context.md: "If tasks.md has a new [x] but state.json cyclesRun hasn't incremented, the previous cycle failed to complete. Increment cyclesRun and proceed."
